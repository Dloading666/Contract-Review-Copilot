# OCR 准确度升级设计

## 目标

把当前“单张图片 -> PaddleOCR -> 用户确认”的链路升级为更稳的文档导入流程，重点解决合同照片识别不准、阅读顺序混乱、截图噪音干扰和多页材料处理能力不足的问题。

升级后的目标链路是：

`文件上传 -> 预处理 -> OCR -> Kimi 纠错 -> 合并文本 -> 用户确认 -> 合同分析`

## 范围

本次设计覆盖以下输入类型：

- `txt`：继续直接读取文本
- `docx`：继续用现有提取逻辑读取文本
- `jpg/jpeg/png/webp`：支持一次选择多张，按选择顺序串行处理
- `pdf`：
  - 可直接提取文本的 PDF：优先直接抽取文本
  - 扫描版 PDF：逐页转图片后进入 OCR 链路

本次设计不包含：

- 前端拖拽排序图片页序
- OCR 后自动开始合同分析
- 切换 OCR 纠错模型

## 核心决策

### 1. 串行处理多张图片

多张合同照片按用户选中的顺序依次处理，不做并行识别。

原因：

- 页序最稳定
- 单页失败时更容易定位问题
- 日志、重试和回退逻辑更清晰

### 2. OCR 纠错固定使用收费模型

OCR 纠错阶段固定使用 `收费模型（kimi）`，不跟随上传区模型选择器变化。

原因：

- 保证 OCR 文本修复质量稳定
- 避免用户误以为上传区模型会影响 OCR 识别精度

上传区模型选择器仍然保留，但仅用于：

- 用户确认后的合同分析
- 后续问答
- 报告生成

### 3. 先确认，再分析

OCR 和纠错完成后，前端只把合并后的文本填入右侧文档区，进入 `ocr_ready` 状态。用户确认或手动修改后，才进入现有 `/api/review` 流程。

原因：

- 拍照合同和扫描件仍可能有错字、漏字、顺序问题
- 用户可以先修正脏数据，避免把错误文本直接送去分析

## 架构设计

### 后端处理链路

新增统一导入入口，例如：

- `POST /api/ocr/ingest`

后端按输入类型分流：

1. `txt`
   - 直接读取文本
2. `docx`
   - 复用现有 docx 抽取逻辑
3. 图片批量上传
   - 按顺序逐张执行：
     - 图片预处理
     - PaddleOCR 初识别
     - Kimi OCR 纠错
4. `pdf`
   - 先尝试直接抽取文本
   - 若文本为空或明显不足，则逐页转图片，再进入同一套 OCR 流程

### OCR 页处理流程

每一页的标准处理顺序：

1. 读取原始图片
2. 图片预处理
3. PaddleOCR 初识别
4. 基于检测框坐标重排阅读顺序
5. 调用 Kimi 对该页 OCR 结果做纠错和格式恢复
6. 输出页级文本、置信度和警告信息

### 合并策略

所有页处理完成后，后端按页序拼接：

- 页与页之间用稳定分隔符或空行分开
- 形成 `merged_text`
- 返回给前端作为右侧文档区的初始文本

## 图片预处理设计

预处理目标是提升原始图像质量，而不是改变合同内容。

首版包含以下步骤：

1. 灰度化
2. 自适应二值化或增强对比度
3. 去噪
4. 倾斜矫正
5. 分辨率不足时放大
6. 尽量裁掉无关边缘区域

预期收益：

- 降低弱光、阴影、压缩噪点对识别的影响
- 提升小字和模糊字的可读性
- 减少拍照歪斜导致的错行和漏字

## Kimi 纠错设计

### 输入

Kimi 不直接重新读图，而是读取页级 OCR 结果进行纠错。

输入内容包括：

- 当前页的 OCR 原始文本
- 可选的低置信度提示
- 明确要求“保留原文含义，不总结、不改写，只修正错字、标点、换行和合同格式”

### 输出

每页返回一段纠错后的纯文本，用于后续拼接。

### 回退

如果 Kimi 纠错失败：

- 保留 PaddleOCR 原始文本
- 在 `warnings[]` 中加入该页提示
- 前端在待确认态提醒用户重点检查该页内容

## PDF 处理设计

### 可提取文本 PDF

先用文本提取方案读取全文。

若结果满足以下条件，则直接使用：

- 文本长度达到最低阈值
- 可读字符比例合理
- 非空页比例足够

### 扫描版 PDF

若直接提取文本失败，或识别结果明显异常，则：

1. 把 PDF 按页转为图片
2. 逐页进入“预处理 -> OCR -> Kimi 纠错”链路
3. 按页序拼接为完整文本

## 接口设计

统一导入接口返回结构建议如下：

```json
{
  "source_type": "image_batch",
  "used_ocr_model": "kimi",
  "merged_text": "完整合同文本",
  "pages": [
    {
      "page_index": 1,
      "filename": "page-1.jpg",
      "text": "第一页文本",
      "average_confidence": 0.97,
      "warnings": []
    }
  ],
  "warnings": []
}
```

字段约定：

- `source_type`：`txt` / `docx` / `image_batch` / `pdf_text` / `pdf_ocr`
- `used_ocr_model`：固定为 `kimi`
- `merged_text`：供前端文档区直接展示
- `pages[]`：保留页级结果，方便后续提示和排错
- `warnings[]`：整体级警告，例如“第 2 页置信度较低”

## 前端交互设计

### 上传区

上传区调整为支持：

- 单个 `txt`
- 单个 `docx`
- 单个 `pdf`
- 多张图片批量选择

首版不支持：

- 混合上传 PDF 和图片
- 前端重新排序页序

### 状态

保留现有 `ocr_ready` 中间态，并新增更清晰的处理提示：

- `uploading`：文件上传中
- `ocr_processing`：正在识别和纠错
- `ocr_ready`：识别完成，等待用户确认

### 识别过程反馈

前端在处理期间显示类似：

- “正在识别第 1/4 页”
- “正在用收费模型修复识别结果”

### 识别完成后

右侧文档区展示 `merged_text`，允许用户：

- 直接编辑文本
- 查看整体警告
- 再点击“确认并开始分析”

## 失败处理

### 单页失败

如果多页里某一页失败：

- 不直接整单失败
- 返回已成功页文本
- 在 `warnings[]` 中标记失败页
- 前端提示用户检查后决定是否重传

### 全部失败

如果所有页都失败：

- 返回错误
- 前端提示重新上传
- 错误文案尽量具体，如分辨率过低、文件损坏、图片过暗

### 低置信度

如果某页平均置信度偏低：

- 保留该页结果
- 把该页标记进警告信息
- 提醒用户在确认前优先检查这些页

- 警告信息在待确认态可见

## 风险与注意事项

- 多页 OCR + Kimi 纠错会增加处理时间，需要前端明确展示处理中状态
- 扫描版 PDF 逐页转图会增加后端 CPU 和内存压力
- Kimi 纠错提示词必须严格限制，只允许“纠错和排版”，不能改写合同内容
- 需要保留页级原始 OCR 结果，便于调试和后续质量优化

## 实施顺序

1. 新增后端统一导入接口和页级结果结构
2. 加入图片预处理模块
3. 接入 Kimi OCR 纠错
4. 支持多图顺序串行处理
5. 增加 PDF 文本提取与扫描件回退
6. 更新前端上传区和待确认态交互
7. 补齐测试并做样本回归验证

# 全站固定 Kimi K2.5 设计

## 目标

将当前前后端仍然保留的“模型选择”能力完全移除，改为整站统一使用 `Kimi K2.5`。

统一范围包括：

- 合同风险扫描
- 合同问答
- 报告生成

本次设计不包含：

- OCR 精度升级链路
- 多图 / PDF OCR 新能力

这些能力仍按各自独立方案推进。

## 用户体验目标

1. 页面上不再显示任何模型选择器
2. 用户上传合同后直接进入处理流程，不需要做模型选择
3. 用户在问答区也不能切换模型
4. 前后端行为一致，不存在“界面固定，后端偷偷切模型”的情况

## 核心决策

### 1. 前端彻底移除模型选择

上传区不再显示模型选择器，也不再保留“免费模型 / 收费模型”的说法。

前端删除内容包括：

- 模型选择 UI
- 模型选项常量
- 模型本地存储逻辑
- `/api/models` 拉取逻辑
- 请求里显式传递 `model` 的逻辑

### 2. 后端统一固定到 Kimi K2.5

后端统一将主模型解析为 `kimi`，并让以下入口都默认使用 `Kimi K2.5`：

- `/api/review`
- `/api/chat`
- 报告生成相关 LLM 流程

保留内部多模型工具函数可以接受，但线上业务入口不再暴露模型分支。

### 3. Docker 配置与运行时保持一致

容器环境里主模型默认值改为 `kimi`，避免出现：

- 本地开发走一个模型
- Docker 里跑另一个模型

## 前端改动

### App.tsx

- 删除 `ModelKey` 联合类型中的其他前端入口使用场景
- 删除 `availableModels` / `selectedModel` 相关状态
- 删除模型同步和本地存储逻辑
- `useStreamingReview` 不再接收前端模型参数
- `/api/chat` 请求体不再发送 `model`

### DocPanel.tsx

- 删除上传区 `ModelSelector`
- 删除模型说明文案
- 上传区只保留文件选择和识别说明

### 测试

- 更新 `App.test.tsx`
- 更新 `DocPanel.test.tsx`
- 确认上传区和问答区都没有模型选择器
- 确认 `/api/chat` 和审查流不再传 `model`

## 后端改动

### llm_client.py

- 将默认模型键改为 `kimi`
- `/api/models` 若继续保留，只返回单一 `kimi`
- 或者允许后续删除该接口，但本次优先做兼容收敛

### main.py

- `/api/chat` 在未传模型时默认使用 `kimi`
- `/api/review` 在未传模型时默认使用 `kimi`
- 如果前端仍意外传来其他模型值，可直接忽略并收敛到 `kimi`

### Docker

- `docker-compose.yml` 中 `PRIMARY_LLM_MODEL_KEY` 改为 `kimi`

## 兼容策略

### 历史前端本地状态

即使用户浏览器里还存有旧的模型选择缓存，也不再读取，也不影响行为。

### 旧接口兼容

后端可以暂时保留 `/api/models`，但只返回单一 `Kimi K2.5`，避免旧代码或测试直接报错。

## 验证

### 前端

- 上传区不再显示模型选择器
- 问答区不再显示模型信息
- 审查请求仍能正常开始
- 问答请求仍能正常返回

### 后端

- `/api/models` 返回单模型或保持兼容
- `/api/chat` 默认使用 `kimi`
- `/api/review` 默认使用 `kimi`

### 部署

- `docker compose up --build -d` 成功
- `http://localhost:3000` 返回 `200`
- `http://localhost:8000/health` 返回正常状态

## 风险

1. 如果 `KIMI_MODEL` 或相关 API key 未配置正确，整站都会受影响
2. 失去前端切换模型的回旋空间，排障时需要改配置而不是点界面
3. 现有测试里如果仍假设有模型切换，需要同步更新

## 实施顺序

1. 前端删除模型选择器和相关状态
2. 后端收敛默认模型为 `kimi`
3. 更新测试
4. 重新构建并部署 Docker
